# RepLog — v1.1 Spec: Live Logger + Scored Coach

**Date:** 2026-06-23
**Status:** Spec (foundation landed; logger UI + coach redesign are the next wave)
**Owner:** Benas Barciauskas (`benasbarciauskas`)
**Builds on:** v1 (`docs/superpowers/specs/2026-06-22-replog-v1.md`, `…-design.md`) and the design
contract in `DESIGN.md`.

## 0. What this spec is

v1 ingested *past* workout notes (screenshots / paste) into a read-only history + analytics +
imbalance coach. v1.1 turns RepLog into a **live workout logger** you train with in real time
, adds **progression-graph views**, and reframes the coach as a **scored
scored experience** (a 0–100 Balance Score with ring gauges + recommendation cards).

This document is the **design contract** for the next wave. The *foundation* — data model,
persistence, hooks, nav, routes, and placeholder pages — is already implemented on
`feat/logger-foundation` (see §6). The next wave builds the actual UIs against that foundation;
it must not change the persisted data model contracts described in §2/§6 without updating this spec.

Hard constraints from v1 still hold: **on-device only, no backend/login/cost, no LLM in the
product, web-first, dark-first, mobile-first, `prefers-reduced-motion` respected.** The visual
language stays the one in `DESIGN.md` — acid-lime accent used sparingly, numbers-as-heroes,
tabular figures, Linear-grade cleanliness with energy.

## 1. Live logger

The core new flow. You log a workout *while training*, set by set.

### 1.1 Starting a workout
- From `/log`: **Start blank workout** or **Start from a routine** (§2). Starting writes an
  `ActiveSession` (one row, persisted) so it survives a refresh / backgrounded PWA.
- If an `ActiveSession` already exists, `/log` resumes it (don't silently discard in-progress work).
- A workout records `startedAt` (drives the elapsed timer), optional `bodyweightKg`, and an optional
  `splitCanonical` (user-picked or inferred from the exercises).

### 1.2 Adding exercises
- **Catalog search** to add exercises (reuse the v1 parser catalog — `createCatalog()` /
  `src/parser/catalog.ts` — for canonical names + aliases + fuzzy match). Adding creates an
  `ActiveExercise` with an empty first set row.
- Uncatalogued lifts are allowed (`exerciseId: 'unknown:<slug>'`, same convention as v1) and may be
  promoted to a custom `ExerciseDef` (existing `addCustomExercise`).
- Per-exercise: a note, a reorderable position, and a per-exercise rest default (`restSeconds`).

### 1.3 Per-set rows
Each `ActiveExercise` shows a table of `ActiveSet` rows. Per row:
- **Set #** (working sets numbered; warm-ups shown but not counted in the working number).
- **"Previous"** — the same exercise's matching set from the last time it was logged, shown as
  `weight × reps` (greyed, tappable to prefill). Derive from saved `Workout`s (last session that
  contains this `exerciseId`), matching by set index.
- **Weight** (in the user's `unit`; stored as kg) and **reps** inputs.
- **RPE** (optional, 6–10, half steps) — see `SetEntry.rpe?` / `ActiveSet.rpe?`.
- **Warm-up toggle** (`isWarmup`) — warm-ups are saved but excluded from working-set counts/PRs.
- **Done check** (`done`) — toggling done is what (a) counts the set toward the workout and
  (b) **auto-starts the rest timer** (§1.4). Only `done` sets are written on finish (§1.6).
- Add / remove / duplicate set rows.

### 1.4 Rest timer
- Auto-starts when a set is marked **done**, counting down from the exercise's `restSeconds`
  (falls back to `AppSettings.defaultRestSeconds`, default 120s).
- Adjustable mid-rest (±15s, or set exact). Pause / skip / reset. Persist the running timer's target
  so a refresh doesn't lose it (store deadline timestamp, not a tick counter).
- Gentle completion cue (respect reduced-motion; no jarring flashes). Optional sound is a later nicety.

### 1.5 Plate calculator
- Given the target weight and `AppSettings` (`barWeightKg` default 20kg, `availablePlatesKg` default
  `[25,20,15,10,5,2.5,1.25]`), compute **plates per side** with a greedy largest-first fill and show a
  visual stack. Surface a remainder when the target isn't reachable with available plates.
- Configurable from Settings: bar weight, available plates, unit. (Settings UI is part of this wave or
  a fast-follow; the `AppSettings` store + defaults already exist.)

### 1.6 Timers, finish, discard
- **Elapsed workout timer** from `startedAt`, always visible while logging.
- **Finish** → a summary (exercises, working sets, total volume, any PRs hit) → **save**.
  Saving calls `repository.finishActiveSession(meta?)` which: converts every `done` set into a
  `SetEntry`, drops exercises with zero done sets, builds a `Workout` (`id` via `newId()`,
  `date` = today local, `createdAt` = now, `dateConfidence: 'high'`, `splitCanonical` from meta →
  session → `'unknown'`, `bodyweightKg` from meta → session → null), persists it via `saveWorkouts`,
  then clears the active session. **This contract is fixed — see §6.**
- **Discard** → `clearActiveSession()` (confirm first; it's destructive).
- A saved logged workout is indistinguishable downstream from an imported one — it flows into the
  same analytics (bests, blocks, bodyweight), coach, exercise detail, dashboard, and history.

### 1.7 Resumability (non-negotiable)
The `ActiveSession` is the single source of truth for in-progress state and is persisted on every
meaningful edit. A refresh, tab close, or PWA relaunch must restore the exact in-progress workout,
including the rest timer's remaining time.

## 2. Routines / templates

- A **Routine** is a reusable template: a name + an ordered list of
  `{ exerciseId, rawName, targetSets, targetReps?, restSeconds? }`.
- **CRUD** at `/routines`: create, edit (rename, add/remove/reorder exercises, set targets/rest),
  delete. Backed by `getRoutines` / `saveRoutine` / `deleteRoutine`.
- **Start a session from a routine** → seeds an `ActiveSession` with the routine's exercises and
  `targetSets` empty set rows each, `restSeconds` carried over, and `routineId` set.
- **Save a finished/active session as a routine** → derive a Routine from the current exercises
  (sets count → `targetSets`, modal reps → `targetReps`).

## 3. Progress views

### 3.1 Exercise detail (`/exercise/:id`, extend existing)
- **Metric toggle**: Heaviest Weight · Estimated 1RM (existing `epley1rm`) · Best-set Volume
  (weight×reps of the top set) · Session Volume (Σ weight×reps that session) · Total Reps.
- **Time-range selector**: 1M · 3M · 6M · 1Y · All (filters the chart + history below).
- **Per-session set history**: each session listed with its sets (weight×reps, RPE, warm-up marked,
  PR badges). Reuse the line-chart card; the selected metric drives the series.

### 3.2 Dashboard (`/`, extend existing)
- **Progress cards**: recent workouts, weekly volume (with trend), current streak, PRs this period.
- **Workout History** entry point → `/history`.

### 3.3 History (`/history`, new)
- Reverse-chronological list of every saved `Workout` (logged or imported), grouped by month.
- Each row → a **workout-detail** view (date, split, bodyweight, every exercise + sets, total volume,
  PRs). Detail can deep-link into the relevant exercise detail.

## 4. Scored coach (`/coach`, redesign)

Reframe the existing imbalance findings into a **scored, premium experience**. The *inputs* don't
change — keep `RATIO_RULES` (`src/coach/ratios.ts`), `analyzeImbalances` / `muscleCoverage`
(`src/coach/analyze.ts`), and `computeBests`. The *presentation* becomes a scored, gauge-based view.

### 4.1 Balance Score (0–100)
- A single headline number: how close the user's lifts sit to the ideal strength ratios.
- **Derivation (deterministic, on-device):** for each `RatioRule` that can fire (both lifts have an
  e1RM), score how far the observed ratio sits inside `[healthyMin, healthyMax]` — full credit when
  inside the band, falling off smoothly outside it (clamp to 0). Weight each rule's contribution by
  `confidence` (high > med > low). Average the weighted per-rule scores → 0–100. Rules that can't fire
  (missing lift pairs) are excluded and surfaced as "unlock more" prompts, not zeros.
- Render as a large central **ring gauge** (numbers-as-heroes), acid-lime fill, with a one-line
  qualitative label (e.g. "Well balanced" / "A few gaps" / "Needs attention").

### 4.2 Sub-scores
- Per-area **ring gauges / percentile bars**: e.g. Push/Pull, Upper/Lower, and per major lift
  (bench, squat, deadlift, OHP). Each derived from the relevant subset of `RATIO_RULES`.
- Subtle gradients + iOS-native feel layered on the current dark + acid-lime system. Premium, calm,
  not gamified-loud.

### 4.3 Recommendation cards
- The existing coach findings, re-presented as **"do this to improve"** cards: the lagging lift, the
  affected muscles, and a concrete prescription (bring lift X up toward the band). Ordered by impact
  (priority → flag). Keep `muscleCoverage` ("never trained" / "undertrained") as a secondary section.

### 4.4 Design references
- Established lifting trackers for the logging + progress-graph patterns.
- Modern scored-card health apps for the score gauges, sub-score rings/percentile bars, recommendation cards, and the
  premium iOS UI feel.

## 5. Navigation (implemented)

- Desktop sidebar lists **all** routes. Mobile bottom nav: four flanking tabs around a **prominent
  centred "Log" action** (the core v1.1 flow), plus a **"More" overflow sheet** for non-primary
  routes — so adding routes never crowds the bar. Thumb-reachable, reduced-motion safe.
- Primary (mobile tabs): Dashboard, Coach, **Log** (centre), History. Overflow: Routines, Import,
  Review. See `src/components/nav-items.ts`.

## 6. Foundation handoff (already implemented on `feat/logger-foundation`)

The next wave builds against these — treat the contracts as fixed.

### 6.1 Types — `src/types/models.ts`
- `SetEntry` extended with **`rpe?: number | null`** (optional; existing data/tests unaffected).
- `ActiveSet { id; weightKg: number|null; reps: number|null; rpe?: number|null; isWarmup?: boolean; done: boolean }`
- `ActiveExercise { id; exerciseId: string; rawName: string; unit: Unit; sets: ActiveSet[]; restSeconds?: number; note?: string }`
- `ActiveSession { id: string; startedAt: string; routineId?: string|null; bodyweightKg?: number|null; splitCanonical?: SplitCanonical; exercises: ActiveExercise[] }`
- `Routine { id; name; exercises: { exerciseId; rawName; targetSets; targetReps?; restSeconds? }[]; createdAt; updatedAt }`
- `AppSettings { barWeightKg: number; availablePlatesKg: number[]; defaultRestSeconds: number; unit: Unit }`

### 6.2 Persistence — `src/data/db.ts`
- Dexie schema bumped to **v2 (additive)**; v1 tables unchanged. New tables:
  `activeSession` (single row, key `'current'`), `routines` (`id, updatedAt`), `settings`
  (single row, key `'app'`). Exported keys: `ACTIVE_SESSION_KEY = 'current'`, `SETTINGS_KEY = 'app'`.

### 6.3 Repository — `src/data/repository.ts`
All existing methods unchanged. New on the `Repository` interface + `LocalRepository`:
- `getActiveSession(): Promise<ActiveSession | undefined>`
- `saveActiveSession(s: ActiveSession): Promise<void>` — persists under the singleton key regardless of the passed `id`.
- `clearActiveSession(): Promise<void>`
- `finishActiveSession(meta?: { bodyweightKg?: number|null; splitCanonical?: SplitCanonical }): Promise<Workout | undefined>`
  — **contract:** done-only sets → `Workout`, drops empty exercises, `id`=`newId()`, `date`=today (local),
  `createdAt`=now, `dateConfidence: 'high'`, split/bodyweight resolved meta → session → fallback
  (`'unknown'` / `null`), saves via `saveWorkouts`, then clears the session; returns the Workout
  (or `undefined` if no active session).
- `getRoutines() / saveRoutine(r) / deleteRoutine(id)`
- `getSettings(): Promise<AppSettings>` (returns `DEFAULT_SETTINGS` when unset; never the storage `id`)
  / `saveSettings(s)`
- `clearAll()` now also clears `activeSession`, `routines`, `settings`.
- Exports: `DEFAULT_SETTINGS` (bar 20kg, plates `[25,20,15,10,5,2.5,1.25]`, rest 120s, kg),
  `stripSettingsKey(stored)`.

### 6.4 Hooks — `src/data/hooks.ts`
All existing hooks unchanged. New live hooks (`useLiveQuery`):
- `useActiveSession(): ActiveSession | undefined`
- `useRoutines(): Routine[]` (newest-updated first)
- `useSettings(): AppSettings` (seeded with `DEFAULT_SETTINGS`)

### 6.5 Routes + placeholder pages — `src/App.tsx`
Code-split via `lazy()`, alongside the existing routes:
- `/log` → `src/features/logger/LoggerPage.tsx`
- `/routines` → `src/features/routines/RoutinesPage.tsx`
- `/history` → `src/features/history/HistoryPage.tsx`

Each is a default-export page using `PageHeader` + a coming-soon `EmptyState`. The `HomeRoute`
loading-guard (count `undefined` → `RouteFallback`) is intact.

### 6.6 Tests
- `src/data/__tests__/repository.logger.test.ts` covers: active-session save/get/clear round-trip;
  singleton-key dedupe; `finishActiveSession` (done-only, empty-exercise drop, meta overrides, split
  fallback, RPE carry, no-session → undefined); routine CRUD; settings defaults + round-trip; and
  `clearAll` clearing the new tables.
- `src/App.test.tsx` adds route-render checks for `/log`, `/routines`, `/history` and the Log nav entry.

## 7. What the next-wave agents must know

- **Logger UI agent:** the `finishActiveSession` contract (§1.6 / §6.3) is the save boundary — build
  the UI to populate `ActiveSession` and call it; don't re-implement Workout construction. Persist
  `ActiveSession` on every edit for resumability (§1.7), including the rest-timer deadline. Use
  `AppSettings` defaults (§6.3) for the plate calculator + rest timer. Reuse the v1 catalog for
  exercise search. Store weights in **kg**; display in `settings.unit` via `formatWeight`.
- **Coach-redesign agent:** keep `RATIO_RULES` + `analyzeImbalances` + `muscleCoverage` as the data
  source (§4) — only the presentation changes. The Balance Score must be deterministic and on-device
  (§4.1). Stay within the `DESIGN.md` token system; acid-lime stays sparing even on gauges.
- **Both:** dark-first, reduced-motion guards everywhere, numbers-as-heroes, no new heavy deps without
  cause (the bottom-nav overflow is hand-rolled — no drawer lib was added).
